## 0. API 문서 (Swagger)

이제 API 명세는 두 가지 방법으로 확인 가능:

**브라우저용 (가장 편함)**
```
https://petstore.swagger.io/?url=https://raw.githubusercontent.com/handyappcrew-bot/handy_th/main/docs/openapi.json
```
백엔드 안 켜져 있어도 바로 명세 확인 가능. 단 "Try it out"은 작동 안 함.

**로컬 (실제 호출 테스트용)**
```
http://localhost:8000/docs
```
백엔드 켜져 있을 때만. 로그인은 `POST /api/auth/dev/login` → `{ "phone": "01012345678" }` 한 번 하면 쿠키 자동 저장됨.

핵심 엔드포인트에는 응답 예시(`Example Value`)가 들어있음. JSON 형태 그대로 참고 가능.

## 1. 신규/변경된 API

### 직원 매장 가입 흐름

#### `POST /api/employee/verify-code` — 매장 코드 조회
- **인증 필수**로 변경 (쿠키 필요)
- 응답에 `membership_status` 필드 추가:
  - `"none"` → 가입 가능
  - `"active"` → 이미 이 매장의 활성 멤버
  - `"pending"` → 가입 신청 대기 중

**프론트에서 처리할 부분 (현재 누락):**
`StoreCodeStep.tsx`의 `handleVerify`에서 응답을 분기해야 함.

```js
if (res.ok) {
  if (data.membership_status === "active") {
    setStatus("error");
    setErrMsg("이미 가입된 매장이에요");
    return;
  }
  if (data.membership_status === "pending") {
    setStatus("error");
    setErrMsg("가입 신청이 진행 중이에요");
    return;
  }
  setStatus("success");
  onCodeVerified(data.id, code, data);
  setTimeout(() => onNext(), 1200);
}
```

#### `POST /api/employee/member/request` — 가입 신청 (안전망)
- 이미 가입된 매장이거나 pending 신청이 있으면 **409 Conflict** 반환
- `detail`에 사유 메시지 포함 (예: "이미 가입된 매장이에요.")

---

### 사장 매장 등록 흐름

#### `GET /api/owner/business/{bno}` — 사업자등록번호 검증
- **인증 필수**로 변경
- 응답에 `registration_status` 필드 추가:
  - `"none"` → 등록 가능
  - `"already_owner"` → 본인이 이미 등록한 매장
  - `"registered_by_other"` → 다른 사장이 이미 등록한 매장
- 국세청 응답 구조 그대로 + 위 필드만 추가
- `data[0].b_stt_cd === "01"` 인 경우만 정상사업자 (그 외는 휴업/폐업/미등록)

#### `POST /api/owner/stores` — 매장 등록
- raw_digits 중복일 경우 **409 Conflict** 반환
- 본인 등록: "이미 등록한 매장이에요."
- 타인 등록: "이미 다른 사장님이 등록한 매장이에요."

---

### 사장 프로필 이미지

#### `POST /api/owner/mypage/{store_id}/profile-image`
- multipart/form-data
- 필드: `store_member_id`(int), `image`(파일)
- 응답: `{ "image_url": "/uploads/profile/xxx.jpg" }`

#### `DELETE /api/owner/mypage/{store_id}/profile-image?store_member_id={id}`
- 응답: `{ "image_url": null }`

### 직원 마이페이지 수정 (겸직 대응)

#### `POST /api/employee/mypage/edit`
- `store_id` 필드 **필수**로 처리됨 (겸직 직원 대응)
- 없을 경우 가게가 2개 이상이면 400 에러
- 이미지 업로드 시 응답에 `image_url` 포함

#### `DELETE /api/employee/mypage/profile-image?store_id={id}` (신규)
- store_id 필수 (겸직 대응)

## 2. 국세청 API 검증

`/api/owner/business/{bno}` 호출 시 응답 해석 방법:

```json
{
  "data": [{
    "b_stt_cd": "01",   // ← 이걸 확인!
    "b_stt": "계속사업자",
    ...
  }],
  "registration_status": "none"
}
```

| `b_stt_cd` | 의미 | 프론트 처리 |
|------------|------|------|
| `"01"` | 계속사업자 | 정상 |
| `"02"` | 휴업자 | "휴업 중인 사업자 번호에요" |
| `"03"` | 폐업자 | "폐업한 사업자 번호에요" |
| `""` (빈 값) | 국세청 미등록 | "등록되지 않은 사업자 번호에요" |

그 다음 `registration_status` 확인해서 추가 처리.

## 4. 매출 조회 (신규)

마감보고(`DailyClosingReport`) 데이터를 집계해서 반환. 별도 매출 테이블 없음.

#### `GET /api/owner/store/{store_id}/sales/monthly?year&month`
월별 매출. 응답:
```json
{
  "year": 2026, "month": 5,
  "daily": { "13": {"net": 405000, "gross": 420000}, ... },
  "total_net": ..., "total_gross": ..., "report_count": ...
}
```

#### `GET /api/owner/store/{store_id}/sales/daily?date=YYYY-MM-DD`
특정 날짜 상세 매출. 마감보고가 없으면 404.
```json
{
  "date": "2026-05-13",
  "card_sales": 200000, "cash_sales": 150000, ...
  "gross_sales": 420000, "net_sales": 405000,
  "cash_shortage": 5000, "manager_note": "이상 없음"
}
```

> 매출 정정은 별도 API 없이 마감보고 수정 API (`PUT .../closing-reports/{id}`) 사용.

## 6. 직원 가입 승인/거절 (신규 엔드포인트)

기획자 프론트 호출 방식에 맞춰 `accept` / `reject` 별도 엔드포인트 추가.

#### `POST /api/owner/store/{store_id}/member-requests/{req_id}/accept`
- 가입 신청 승인
- 응답: `{ "ok": true }`
- 승인 시 자동으로 `StoreMembers` + `StaffContract` 레코드 생성

#### `POST /api/owner/store/{store_id}/member-requests/{req_id}/reject`
- 가입 신청 거절
- 응답: `{ "ok": true }`

기존 `PUT /member-requests/{req_id}` (body `{status}`)은 레거시로 유지.

## 7. 비밀번호 찾기/재설정 (신규)

3단계 플로우. 회원가입 SMS와 동일한 패턴이지만 분리된 엔드포인트.

#### `POST /api/auth/password/reset/code/send`
- 요청: `{ "phone": "01012345678" }`
- 가입된 번호인지 확인 후 SMS 발송. 가입 안 된 번호면 404.
- 개발 환경: 인증번호는 항상 `00000` (실제 발송 X)

#### `POST /api/auth/password/reset/code/verify`
- 요청: `{ "phone": "01012345678", "code": "12345" }`
- 인증 성공 시 **`password_reset_token` 쿠키 발급** (10분 유효, httpOnly)
- 이 쿠키는 다음 단계에서 자동으로 전송됨 (프론트는 `credentials: 'include'` 필요)

#### `POST /api/auth/password/reset`
- 요청: `{ "phone": "01012345678", "new_password": "newPass123" }`
- 쿠키의 `password_reset_token` 자동 검증
- 만료/누락 시 401 → 다시 인증부터
- 비밀번호 정책: 8~16자, 영문+숫자 포함
- 성공 시 쿠키 삭제

**프론트 흐름**:
```
1. 전화번호 입력 → POST /password/reset/code/send
2. 인증번호 입력 → POST /password/reset/code/verify (쿠키 자동 발급)
3. 새 비밀번호 입력 → POST /password/reset (쿠키 자동 전송)
4. 성공 → 로그인 화면
```

> ⚠️ 프론트에서 fetch 호출 시 `credentials: 'include'` 반드시 포함해야 쿠키가 자동 전달됨.

**보안 포인트 (참고용)**:
- 토큰 타입을 `"password_reset"`으로 명시 → access_token이나 signup_token으로 비밀번호 재설정 시도 불가
- 토큰 10분 만료 → 인증 후 시간 지체되면 다시 인증부터
- httpOnly 쿠키 → JavaScript 접근 불가 (XSS 대응)
- 비밀번호 변경 성공 직후 쿠키 즉시 삭제
- SMS 발송 시 가입된 번호인지 확인 → 가입되지 않은 번호로는 재설정 불가
- 하루 인증번호 발송 5회 제한 (기존 회원가입 SMS와 동일 정책)

## 8. 알림 전체 읽음 처리 (신규)

#### `PATCH /api/common/notification/read-all?store_id={id}`
- 해당 매장 내 본인 알림 중 안 읽은 것들을 모두 읽음 처리
- 응답: `{ "success": true, "updated": 7 }` (updated = 변경된 알림 개수)
- 알림 목록 화면 상단 "모두 읽음" 같은 버튼에 연결 용도. 필요 없으면 그냥 사용 안 하면 됨.

## 9. 직원 휴직/퇴직 처리 — 기획 협의 필요 ⏸️

> 백엔드 구현 보류. 기획자와 함께 결정 후 진행 예정.

### 현재 데이터 모델 (이미 사용 가능)
- `StaffContract.working_status` 컬럼 존재 (기본값 `"재직"`, 주석에 "재직 / 퇴직")
- `StoreMembers.is_deleted` (소프트 삭제용)
- → **DB 마이그레이션 없이도 처리 가능**

### 백엔드가 생각한 단순 설계 (MVP)
| 액션 | 엔드포인트 | 동작 |
|------|-----------|------|
| 휴직 | `POST /api/owner/store/{store_id}/staff/{staff_id}/leave` | `working_status = "휴직"` |
| 복직 | `POST /api/owner/store/{store_id}/staff/{staff_id}/return` | `working_status = "재직"` |
| 퇴직 | `DELETE /api/owner/store/{store_id}/staff/{staff_id}` | `is_deleted = True`, `working_status = "퇴직"` |

### 기획자랑 같이 정해야 할 의문점

1. **휴직 시 추가 정보 입력 받을지?**
   - 휴직 사유?
   - 휴직 시작 날짜 / 복귀 예정 날짜?
   - 휴직 메모?
   → DB 컬럼 추가 필요할 수도 있음

2. **퇴직 시 추가 정보?**
   - 퇴직 사유?
   - 퇴직 일자(자동 = 오늘 vs 사장 지정)?
   - 마지막 근무일?

3. **휴직 중 동작 정책**
   - 새로운 스케줄 등록 막을지? (UI에서 휴직자 선택 비활성)
   - 출근 기능 막을지? (휴직자가 앱에서 출근 못 하게)
   - 알림 받을 수 있게 둘지?
   - 휴직 기간 동안 급여명세서 발행 가능?

4. **퇴직 후 데이터 표시**
   - 직원 목록에서 퇴직자 안 보이게? 별도 탭으로 보이게?
   - 과거 급여명세서, 출근 기록은 사장 측에서 계속 조회 가능?
   - 퇴직 직원이 본인 앱에서 과거 급여명세서 조회 가능?

5. **누가 트리거하나?**
   - 옵션 A: 사장이 일방적으로 처리 (단순)
   - 옵션 B: 직원이 신청 → 사장 승인 (가입 신청 패턴과 일관성)
   - 옵션 C: 둘 다 가능

6. **복귀 가능 여부**
   - 퇴직했다가 다시 가입 가능?
     - is_deleted=True 인 StoreMembers가 있을 때 어떻게 처리?
     - 이전 데이터 복구 vs 새로 생성?

7. **`working_status` 값 종류**
   - 현재 모델 주석: "재직 / 퇴직"
   - 실제 사용에선 "휴직"도 필요
   - 가능한 값을 명확히 정해야 함: `재직 | 휴직 | 퇴직` ?
   - 또는 별도 enum 컬럼 필요?

### 보류 사유
백엔드 구현은 매우 단순하지만, 위 정책 결정이 안 되면 나중에 뒤집어야 함. 기획자와 30분 정도 미팅으로 정리 후 진행 권장.

## 10. 협업 흐름

```
기획자: Swagger의 request body + 사용 화면 + 기대 동작을 전달
   ↓
본인: 기획자 요청에 맞춰 백엔드 API 수정
   ↓
git push → 기획자 pull → 백엔드 재시작 → 동작 확인
```

기획자가 Swagger 명세 보고 프론트에서 호출 후, 응답 형태가 안 맞으면 알려주면 백엔드 쪽 조정.
